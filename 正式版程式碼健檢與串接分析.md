# 莎莎保險助理工作台：正式版程式碼健檢與串接分析

檢查日期：2026 年 6 月 12 日  
檢查範圍：`index.html`、`app.js`、`styles.css`、`data/*`、`README.md`、`package.json`、`start.bat`

## 一、結論

目前版本適合作為介面展示與需求確認原型，不適合直接存放或處理正式客戶資料。

主要原因不是畫面，而是底層仍缺少正式後端、資料關聯、輸入安全、錯誤處理、測試、部署、監控、備份及多裝置同步。正式版不應只把現有網頁直接放到網路上，而應保留目前的視覺設計，重新建立正式的資料與服務架構。

## 二、已完成的檢查

- JavaScript 語法檢查：`app.js`、`daily-quote.js`、`weekly-insights.js` 均通過。
- HTML 識別碼檢查：共 111 個 `id`，沒有重複。
- JavaScript 畫面元件引用：共 101 個不同引用，沒有找不到的 `id`。
- 檔案編碼檢查：主要程式均為有效 UTF-8，沒有損壞字元。
- 檢查所有 `localStorage`、`innerHTML`、檔案上傳、外部網址、表單、對話框及主要操作事件。
- 專案目前沒有自動化測試、程式碼檢查、正式建置或部署指令。

本次無法完成完整的瀏覽器自動化操作測試，因此正式開發時仍須補上桌機與手機的端對端測試。

## 三、最高優先問題

### P0-1：使用者輸入可能形成儲存型網頁注入

下列位置會把使用者輸入或未來外部資料直接放入 `innerHTML`：

- `app.js:147`：通知訊息直接寫入 HTML。
- `app.js:250-260`：行程名稱及地點直接組成今日時間軸。
- `app.js:337-340`：保單欄位直接組成表格。
- `app.js:1046-1082`：新聞、法規及網址直接組成畫面。

例如行程名稱若包含惡意 HTML，儲存後可能在重新顯示時執行。正式版若有多人共用、登入憑證或客戶資料，影響會更嚴重。

處理方式：

- 一般文字改用 `textContent` 或安全的元件模板。
- 需要 HTML 時使用經核准的消毒函式庫。
- 外部網址只允許 `https:`，並建立來源網域允許清單。
- 加上 CSP 與 Trusted Types 作為第二層防護。

### P0-2：保單畫面顯示「已保存」，實際只保存部分欄位

`index.html:392-399` 提供保單號碼、契約始期、繳費年期及保障摘要，但 `app.js:424-435` 實際只保存：

- 客戶姓名
- 保險公司
- 險種
- 保額
- 年繳保費
- 更新日期

因此使用者校正過的保單號碼、契約日期、繳費年期與保障摘要會遺失。這是正式版前必須修正的資料遺失問題。

### P0-3：客戶、保單、行程與團隊資料沒有真正關聯

目前各功能只是數個獨立陣列：

- 行程使用文字記錄客戶名稱，沒有 `customerId`。
- 保單沒有 `customerId`，也沒有保單明細或附約資料表。
- 客戶的保單數量是人工輸入，不會依保單資料自動計算。
- 刪除客戶不會處理其保單、行程或服務紀錄。
- 移除團隊成員後，客戶負責人及團隊任務可能成為無人負責資料。
- 團隊任務的負責人與期限是自由文字，無法可靠統計。

正式版應先完成資料模型及關聯規則，再串接其他服務。

## 四、高優先問題

### P1-1：瀏覽器資料損壞時可能導致整個程式無法啟動

`app.js:73`、`175`、`333`、`535-537`、`689` 直接使用 `JSON.parse(localStorage...)`，沒有例外處理。

只要其中一筆瀏覽器資料格式損壞，後續畫面初始化就可能中止。也沒有資料版本、格式遷移、容量不足處理、備份或還原機制。

### P1-2：多個按鈕目前只有動畫，沒有真正執行工作

- `app.js:449-458`：「立即更新情報」只有等待動畫，沒有取得新資料。
- `app.js:965-973`：「檢查連結狀態」沒有真的檢查網址。
- `app.js:917-953`：不同文件按鈕都開啟同一家公司的首頁，不是實際下載頁。
- `app.js:1017`：設定中心仍是預留訊息。
- `index.html:316-317`：行事曆週檢視及日檢視停用。

正式版不可在沒有完成工作的情況下顯示成功訊息，否則容易讓使用者誤判資料已更新或已檢查。

### P1-3：行事曆與客戶功能缺少正式工作流程

- 行程只能新增，不能編輯、刪除、拖曳改期或設定重複規則。
- 提醒選項雖有保存，但目前沒有實際作用。
- 沒有會談完成、取消、未出席及後續任務狀態。
- 客戶刪除只有一次確認，沒有回收筒、復原或刪除影響提示。
- 沒有服務歷程、聯繫紀錄、附件、標籤、家庭關係及需求版本。
- 全域搜尋只找第一筆符合資料，沒有完整結果頁與進階條件。

### P1-4：檔案上傳只檢查瀏覽器提供的 MIME 類型

`app.js:348-365` 只檢查 `file.type` 及檔案大小。正式版還需要伺服器端檔案簽章檢查、重新命名、圖片重新編碼、移除中繼資料、病毒掃描、隔離區及安全物件儲存。

這一項即使 OCR 服務另行處理，也仍是獨立且必要的上傳安全功能。

### P1-5：沒有正式網站安全標頭與部署環境

目前由 `start.bat` 直接開啟 `index.html`，沒有：

- HTTPS、正式網域及憑證管理
- Content Security Policy
- HSTS、Referrer Policy、Permissions Policy
- 防點擊劫持設定
- 開發、測試、正式環境區分
- API 金鑰及環境變數管理
- 版本發布、回滾及維護模式

## 五、中優先問題

### P2-1：手機版不是可安裝的正式 PWA

目前沒有 Web App Manifest、App 圖示、Service Worker、離線頁面、快取更新策略及版本更新提示。

手機雖可用「加入主畫面」，但仍只是一般網站捷徑，不具完整安裝、離線及更新管理能力。

### P2-2：無障礙及手機可讀性需要改善

- 多處字體只有 6 至 10px，手機閱讀負擔高。
- 只有輸入欄位設定焦點樣式，多數按鈕缺少清楚的鍵盤焦點提示。
- 日曆日期格使用可點擊的 `div`，鍵盤無法操作。
- 對話框沒有完整焦點鎖定、關閉後焦點還原及背景不可操作處理。
- 商品比較對話框缺少對話框標題關聯。
- `app.js:912-914` 點擊商品比較背景關閉時，沒有同步更新 `aria-hidden`。
- 手機版會隱藏通知鈴鐺及全域搜尋，沒有替代入口。
- 手機行事曆把事件文字改成圓點，無法直接辨識事件內容。

### P2-3：日期及長時間開啟狀態不可靠

- `app.js:41` 只在網頁開啟時取得一次日期，跨午夜不會自動換日。
- `app.js:456` 手動更新情報時仍寫死「6 月 11 日」。
- 每日語錄、商品日期及多項畫面文字是靜態資料。

### P2-4：程式維護與品質管控不足

- `app.js` 共 1,113 行，所有功能集中在同一檔案。
- 沒有 TypeScript、資料結構驗證或統一 API 層。
- `package.json` 沒有依賴、測試、建置、格式化或程式檢查指令。
- 沒有單元測試、整合測試、手機端測試及安全測試。
- 沒有錯誤追蹤、效能監控或系統健康檢查。
- 專案根目錄有 `莎`、`曜` 兩個無內容檔案，應確認後移除。

## 六、除了已列五項外，正式版仍需要的串接與功能

以下不重複 OCR、新聞與法規排程、帳號權限與加密資料庫、外部通知、法遵及個資政策。

### A. 正式網站與手機安裝能力

必要程度：最高

- 正式網域、DNS、HTTPS 憑證。
- 雲端網站託管、CDN、WAF 及流量限制。
- PWA Manifest、App 圖示、Service Worker、離線頁面及更新提示。
- 手機安裝教學與版本相容性提示。

### B. 後端 API 與整合閘道

必要程度：最高

- 前端不可直接連接所有第三方服務，應由後端 API 統一管理。
- 統一處理驗證、重試、逾時、流量限制、去重及錯誤回報。
- 建立 Webhook 接收器，供未來外部系統回傳狀態。
- 每一種保險公司或外部服務使用獨立轉接器，避免更換供應商時重寫整個前端。

### C. 多裝置同步與衝突處理

必要程度：最高

- 手機、平板及電腦即時同步。
- 離線編輯後重新連線的同步佇列。
- 同一筆客戶資料被多人修改時的版本檢查及衝突提示。
- 自動保存、版本歷程及誤刪復原。

### D. 安全檔案儲存與附件服務

必要程度：最高

- 雲端物件儲存及有期限的安全下載網址。
- 檔案格式、簽章及大小驗證。
- 圖片重新編碼、EXIF 等中繼資料移除。
- 病毒掃描、隔離區及失敗重試。
- 客戶、保單、會談與文件附件的權限關聯。

### E. 客戶及保單資料匯入、匯出與去重

必要程度：高

- Excel／CSV 批次匯入與欄位對應。
- Excel、PDF 及列印報表匯出。
- 手機聯絡人或 vCard 匯入。
- 電話、Email、姓名及證件識別資料的重複客戶偵測。
- 匯入前預覽、錯誤報告及可復原機制。

### F. 報表與文件產生服務

必要程度：高

- 客戶保障總覽 PDF。
- 保單清單、保費及到期日報表。
- 商品比較表及版本日期。
- 會談紀錄、服務摘要與團隊進度報表。
- 公司版型、浮水印、頁碼及下載紀錄。

### G. 電子簽署、同意確認與保險公司入口

必要程度：依實際業務流程決定

- 電子簽署服務或保險公司電子要保流程的安全跳轉。
- 文件簽署狀態、版本及完成證明。
- 各公司業務入口的 SSO、深層連結或經授權的轉接方式。
- 若保險公司沒有正式 API，不應自行模擬登入；應使用官方授權方式。

### H. AI 語錄及行動建議服務

必要程度：中高

每日語錄除了排程，還需要：

- 經典書籍語料庫及篇章來源。
- AI 模型 API。
- 引文去重、出處核對及禁止杜撰規則。
- 產生內容的版本、審核及回退機制。
- 今日行動建議的規則引擎，避免只產生空泛文字。

### I. 全文搜尋與資料索引

必要程度：中高

- 客戶、保單、行程、服務紀錄及文件的統一搜尋。
- 注音、同音字、電話格式及模糊搜尋。
- 權限過濾後再顯示搜尋結果。
- 大量資料時可使用專用搜尋服務或資料庫全文索引。

### J. 系統監控、錯誤追蹤及操作稽核

必要程度：最高

- 前端錯誤追蹤及 API 錯誤告警。
- 服務可用率、回應時間、排程成功率及第三方 API 狀態。
- 新增、修改、刪除、匯出、下載及重要查詢的操作紀錄。
- 日誌需避免直接記錄完整客戶敏感資料。
- 系統狀態頁及客服查詢用追蹤編號。

### K. 備份、還原及災難復原

必要程度：最高

- 自動備份、跨區備份及定期還原演練。
- 單筆資料版本還原與整批災難還原。
- 明確定義可接受的資料損失時間與服務恢復時間。
- 備份本身也需加密及控管存取。

### L. 後台管理與資料品質中心

必要程度：高

- 資料來源健康狀態。
- 失敗排程與失敗檔案重試。
- 商品、文件、語錄及內容的人工審核與發布。
- 重複客戶、缺少欄位及過期資料報告。
- 功能開關、公告及維護模式。

### M. 自動測試、發布與供應鏈安全

必要程度：最高

- 單元測試、API 整合測試及手機端端對端測試。
- 程式格式、型別、弱點及依賴套件掃描。
- 測試環境、預備環境及正式環境。
- 自動部署、資料庫遷移、回滾及版本紀錄。
- 第三方套件清單及更新機制。

### N. 商業營運功能

若未來提供其他經理或團隊使用，還需：

- 組織申請、方案、授權席次及帳單。
- 使用量限制及方案升降級。
- 使用者教學、客服、問題回報及服務狀態頁。
- 客戶資料搬出及停用帳號後的交接流程。

## 七、建議正式架構

```text
手機／電腦 PWA
    ↓
後端 API／整合閘道
    ├─ 客戶、保單、行程、團隊服務
    ├─ 搜尋與報表服務
    ├─ AI 語錄／行動建議服務
    ├─ 外部服務轉接器與 Webhook
    └─ 稽核、監控與錯誤追蹤
         ↓
關聯式資料庫＋物件儲存＋快取／工作佇列＋備份
```

前端建議改為可維護的 TypeScript 元件架構；現有配色、版面及操作流程可以保留。資料的新增、修改、刪除、搜尋及同步全部改走後端 API，不再以瀏覽器 `localStorage` 作為主要資料庫。

## 八、建議開發順序

### 第一階段：先讓底層可承載正式資料

1. 定義客戶、保單、附約、行程、服務紀錄、團隊與文件的資料模型。
2. 建立後端 API、正式資料庫、物件儲存及同步機制。
3. 修正網頁注入風險、輸入驗證與網址驗證。
4. 建立測試、正式部署、安全標頭、監控及備份。
5. 將網站改成 HTTPS PWA。

### 第二階段：完成核心工作流程

1. 客戶與保單真正關聯。
2. 行程可編輯、刪除、完成、重複及建立後續任務。
3. 客戶服務歷程、附件、標籤、家庭關係及需求版本。
4. 全文搜尋、Excel 匯入匯出及 PDF 報表。
5. 後台管理、操作紀錄及資料品質檢查。

### 第三階段：再接外部服務

1. OCR、新聞法規商品、外部行事曆及通知。
2. AI 語錄、行動建議與內容審核。
3. 保險公司入口、電子簽署及授權文件流程。
4. 商業方案、客服及營運分析。

## 九、正式上線最低門檻

- 不再使用瀏覽器本機資料作為唯一正式資料來源。
- 所有使用者輸入及外部資料都經過安全輸出處理。
- 客戶、保單、行程與團隊資料有正式關聯及刪除規則。
- 手機與電腦同步並能處理衝突。
- 有備份還原演練、錯誤告警及操作稽核。
- 有自動化測試及正式發布／回滾流程。
- 所有顯示「成功」「已更新」「連結可用」的功能都真的完成驗證。
- 手機版通過實機測試、無障礙檢查及主要瀏覽器測試。
