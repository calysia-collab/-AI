## 三、正式上線阻擋項目

下列項目完成前，不建議輸入真實客戶個資或公開上線。

### P0-1 客戶個資仍可能以明文留在瀏覽器

`app.js` 目前會把客戶、保單、行事曆及團隊資料寫入 `localStorage`。登出雖會清除，但登入期間、瀏覽器異常結束或電腦帳號遭存取時，資料仍可能被直接讀取。

正式版應改為：

- 伺服器資料庫為唯一正式來源。
- 瀏覽器只保留必要且短效的狀態。
- 若需要離線功能，改用加密 IndexedDB、明確到期時間及登出清除。
- 通知已讀狀態依組織及使用者儲存在伺服器，不使用全機共用的本機鍵值。

### P0-2 資料庫敏感欄位尚未欄位級加密

電話、Email、生日、保單號碼、保額、保費、保障摘要、活動內容及備註目前可在資料庫中以明文存在。密碼已安全雜湊，MFA 密鑰也有加密，但客戶資料仍需補強。

正式版應加入：

- KMS 或雲端金鑰管理服務。
- 信封式加密、金鑰版本及輪替。
- 備份加密。
- 必要搜尋欄位的雜湊索引或代碼化設計。
- 金鑰遺失、撤銷及輪替演練。

### P0-3 OCR 上傳尚不是安全文件流程

目前拍照區主要是圖片預覽與模擬辨識進度，沒有真正送到 OCR 後端，也沒有安全附件服務。

正式版必須先完成：

- 私有物件儲存空間。
- 短效上傳及下載網址。
- 檔案簽章與實際內容格式檢查，不能只看副檔名或瀏覽器 MIME。
- 圖片重新編碼、移除 EXIF、檔案大小與頁數限制。
- 惡意程式掃描、隔離區及清除政策。
- OCR 欄位信心分數、人工逐欄確認及原圖對照。
- 辨識失敗重試、重複文件偵測及完整稽核紀錄。

### P0-4 PostgreSQL 尚未真實驗證，移轉檔需先固定

原始檢查時 PostgreSQL 僅通過模擬連線測試，且初始 schema 有被修改後再疊加遷移的風險。階段 0 已將初始結構凍結為 `database/migrations/001_initial_schema.sql`，並以 `database/migrations.lock.json` 驗證每一版遷移檔 checksum；之後不得回頭修改已發布的遷移檔。

正式版應完成：

- 移轉檔一旦發布後不可修改。
- 確認 `001` 的正式基準版本，後續變更只新增 `002`、`003`。
- 使用真實 PostgreSQL 執行全新建庫、連續升級、失敗回滾及重複執行測試。
- 啟用連線加密、憑證驗證、最小權限資料庫帳號。
- 建立自動備份、時間點還原及實際還原演練。

### P0-5 整包狀態同步具有大量覆寫風險

`PUT /api/state` 允許 owner 或 manager 一次取代客戶、保單、行事曆、團隊及任務資料。雖有版本衝突保護，但舊的本機狀態仍可能形成大範圍誤覆寫。

正式版應改為：

- 客戶、保單、行事曆、團隊、任務各自使用單筆或批次 API。
- 每筆資料有版本、更新者、更新時間及衝突處理畫面。
- 批次匯入採工作佇列、預覽、確認及可回復設計。
- `/api/state` 僅保留給受控移轉工具，或正式移除。

### P0-6 尚無正式部署與供應鏈基線

正式版需要：

- 固定 Node.js LTS 版本。
- 建立 `package-lock.json`，固定第三方套件版本。
- CI 自動執行語法、測試、弱點、授權及秘密掃描。
- 測試、預備及正式三個環境。
- HTTPS 網域、反向代理、HSTS、WAF、流量限制及可信代理設定。
- Secrets Manager，不將密碼或金鑰放入檔案。
- 自動部署、資料庫移轉閘門、回滾及維護模式。

- 重複事件、日／週／月檢視。
- Google、Microsoft 或 CalDAV 行事曆串接。
- Email、LINE、Web Push 或手機推播。
- 通知送達、失敗、重試、取消及已讀紀錄。
- 續期、生日、追蹤及服務週期的自動規則。

### 3. 情報週報與法規更新

目前內容仍來自本機靜態資料，更新按鈕沒有真正抓取網路資料。

正式版需要：

- 只使用合法且穩定的官方 API、RSS 或授權資料來源。
- 排程器、工作佇列、重試、去重及來源健康監控。
- 保存來源網址、發布日期、抓取日期及內容版本。
- AI 摘要須保留原始資料引用，不得把推論當成法規事實。
- 發布前人工審核、修正、撤回及更正流程。
- 法規異動的生效日、影響對象、內部待辦及完成狀態。

### 4. 商品比較與公司文件中心

目前六筆商品是示範資料，公司文件連結多數只連到首頁，連結檢查也是畫面模擬。

正式版需要：

- 商品、條款、費率、核保限制及版本化資料模型。
- 官方來源、適用日期、停售日期及最後人工核對時間。
- 比較欄位定義與不可比較項目提示。
- 失效連結自動檢查及人工修正工作台。
- 商品資料變更通知及歷史版本。
- 不模擬登入保險公司後台；需使用業者正式 API、深層連結、SSO 或授權整合。

### 5. 每日古典語錄

目前語錄是本機固定檔案，不是每日 08:00 自動產生。

正式版需要：

- 經典書籍及可信版本來源清單。
- 原文、人物、著作、白話意譯、AI 精解及來源網址。
- 每日 08:00 排程、避免重複、失敗備援及時區管理。
- 來源校對與內容審核，避免誤引、錯別字或杜撰。
- 保留已發布歷史及人工撤換功能。

### 6. 團隊與帳號

目前團隊成員資料與登入帳號是兩套資料，可能出現同一人資料不一致。

正式版應：

- 團隊成員連結唯一 `user_id`。
- 邀請、停用、離職交接、權限變更及客戶重新指派。
- 權限申請與核准紀錄。
- 裝置及 Session 管理。
- 遠端密碼復原使用已驗證 Email、企業身分系統或管理者流程。

## 六、建議開發順序

### 階段 0：固定正式基線

工作內容：

- 建立版本庫與 CI。
- 固定 Node.js、套件 lockfile 及環境設定。
- 整理並凍結 PostgreSQL 移轉檔。
- 建立測試、預備、正式環境。

完成標準：

- 每次提交都會自動測試。
- 全新資料庫與既有資料庫都能安全升級。
- 可一鍵回滾應用程式，資料庫有明確回復程序。

### 階段 1：個資、檔案與正式資料庫

工作內容：

- PostgreSQL 實機串接。
- KMS、欄位加密、備份加密。
- 移除敏感資料明文 `localStorage`。
- 私有附件儲存、惡意檔案掃描及簽名網址。

完成標準：

- 資料庫或備份檔單獨外洩時，敏感欄位不可直接閱讀。
- 登出或 Session 到期後，瀏覽器沒有可讀取的客戶資料。
- 附件只能由有權限的使用者在短效時間內取得。

### 階段 2：資料模型與 API 正式化

工作內容：

- 正規化客戶、保單、服務歷程、團隊及通知資料。
- 以個別資源 API 取代整包 `/api/state` 覆寫。
- 加入分頁、搜尋、匯入匯出、軟刪除及衝突處理。

完成標準：

- 任何單筆修改不會覆寫其他資料。
- 多人同時編輯有明確衝突提示與回復方式。
- 大量資料仍符合回應時間目標。

### 階段 3：OCR 正式流程

工作內容：

- 安全上傳、OCR 供應商、欄位映射、信心分數及人工確認。
- 原圖、辨識結果、修正紀錄及保單建立流程串接。

完成標準：

- 未經人工確認的辨識結果不得直接成為正式保單資料。
- 每個欄位都能追溯原圖、辨識值、修正者及時間。

### 階段 4：情報、法規、商品與文件資料平台

工作內容：

- 官方來源串接、排程、去重、摘要、人工審核及發布。
- 商品與文件版本管理、來源健康及失效連結檢查。
- 每日古典語錄排程與來源校對。

完成標準：

- 每筆內容都有來源、日期、版本、審核者及更新狀態。
- 排程失敗會警示，錯誤內容可撤回並保留更正紀錄。

### 階段 5：行事曆與通知

工作內容：

- 外部行事曆、Email、LINE 或 Push。
- 通知規則、重試、送達、已讀及退訂。

完成標準：

- 可證明提醒是否建立、送達、失敗、重試及被閱讀。
- 不同帳號及組織的通知狀態完全隔離。

### 階段 6：正式上線驗收

工作內容：

- 瀏覽器、手機、無障礙、效能、壓力及安全測試。
- 監控、告警、備份、災難復原、客服及事件處理演練。
- 法遵、個資、保存與刪除政策正式核准。
- 小規模試用、問題修正及分批開放。

完成標準：

- 所有 P0 缺口關閉。
- 沒有未處理的重大或高風險安全問題。
- 備份可在目標時間內還原。
- 監控及告警能在故障時通知負責人。
- 正式環境完成使用者驗收及上線簽核。

## 八、最優先的下一步

建議先完成「階段 0 與階段 1」，不要先投入更多示範畫面或 AI 功能。原因是這兩階段會決定資料庫、加密、附件與部署架構；若先做 OCR、商品或情報串接，之後很可能需要重新搬移資料與重寫權限流程。
