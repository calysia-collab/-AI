# 莎莎保險助理工作台營運維護手冊

## 首次建立主人帳號

1. 在工作台主機上執行 `start.bat`。
2. 開啟 `http://127.0.0.1:4173`。
3. 在首次設定畫面確認工作台名稱與主人名稱「張經理」。
4. 設定登入帳號與至少 12 個字元的強密碼。
5. 密碼不要放在文字檔、聊天訊息或程式碼中。

首次設定只允許從本機進行，且建立完成後不可再次呼叫設定功能。登入後可從個人安全設定自行變更密碼或啟用 MFA。若主人同時遺失密碼、驗證器與復原碼，目前仍須由授權維護人員依復原程序處理。

## 個人密碼與 MFA

1. 登入後按右下角個人設定按鈕。
2. 變更密碼時需輸入目前密碼；成功後其他裝置與舊工作階段會立即登出。
3. 啟用 MFA 時，將畫面上的秘密加入支援 TOTP 的驗證器 App。
4. 先下載或抄錄 10 組復原碼，再輸入驗證器的 6 位數代碼完成啟用。
5. 每組復原碼只能登入一次。使用後應在安全紀錄中標示已失效。
6. 停用 MFA 需同時輸入目前密碼與驗證碼或尚未使用的復原碼，完成後所有工作階段會登出。

剛用來啟用 MFA 的 6 位數代碼不可立即重複用於登入；請等待驗證器顯示下一組代碼。

## 緊急帳號復原

此流程只供已授權、可登入工作台主機的維護人員使用，不會開放未登入的網頁重設入口。

1. 先停止莎莎工作台伺服器。
2. 先預覽帳號，不會修改資料：

```text
npm run account:recover -- --username=登入帳號
```

3. 在 PowerShell 工作階段暫時設定新密碼，密碼不要直接寫在命令列參數：

```text
$env:SASHA_RECOVERY_PASSWORD='新的強密碼'
```

4. 正式執行：

```text
npm run account:recover -- --username=登入帳號 --confirm
```

5. 若密碼、驗證器及 MFA 復原碼同時遺失，才加入：

```text
--disable-mfa
```

6. 完成後立即清除暫時環境變數：

```text
Remove-Item Env:SASHA_RECOVERY_PASSWORD
```

復原會拒絕停用中的帳號與沿用舊密碼，並清除登入鎖定、使所有舊工作階段失效。使用 `--disable-mfa` 時會同步刪除舊 MFA 秘密及復原碼。稽核紀錄會留下 `emergency_recovery`，但不保存密碼。不要把 `SASHA_RECOVERY_PASSWORD` 寫入 `.env`、批次檔、聊天訊息或版本控制。

## 登入安全事件

- 連續登入失敗 5 次會鎖定 15 分鐘。
- 登入閒置超過 30 分鐘，畫面會要求重新登入。
- 發現可疑存取時，先停止服務、備份資料庫，再檢查 `/api/audit`。
- `/api/metrics` 與 `/api/audit` 僅限主人與管理者角色。
- 正式遠端部署必須由 HTTPS 反向代理提供連線，不可直接使用不安全的遠端開放參數。

## 客戶負責人與資料範圍

- 主人與管理者可在客戶資料中選擇負責顧問。
- 顧問新增客戶時，系統會由後端自動指派給目前登入者，不能代替其他顧問建立或轉派。
- 顧問只會取得自己負責的客戶、相關保單及客戶行程。
- 未綁定客戶且分類為「團隊事項」的行程可由顧問共同查看。
- 唯讀成員可查看組織資料，但所有新增、修改及刪除操作都會被拒絕。
- 停用顧問帳號前，主人應先檢查並轉派其客戶；停用不會自動刪除或改派客戶。
- 舊資料若只有顧問姓名，系統只在同組織內恰好有一個同名有效帳號時自動建立關聯。

## 一、啟動與停止

### 啟動

雙擊 `start.bat`，或使用 Node.js 22.5 以上版本執行：

```text
node server.mjs
```

預設網址：

```text
http://127.0.0.1:4173
```

系統已完成登入、角色權限與組織隔離，但伺服器仍只允許本機直接連線。正式環境應由 HTTPS 反向代理連入 `127.0.0.1`，不要直接公開 Node.js 連接埠，也不要以 `SASHA_ALLOW_INSECURE_REMOTE=true` 對外提供服務。

### 停止

關閉「莎莎工作台伺服器」視窗，或在執行視窗按 `Ctrl+C`。停止後再進行資料庫還原。

## 二、系統健康檢查

### 健康狀態

```text
GET /api/health
```

會回傳：

- 系統是否正常
- 資料庫是否可用
- 目前資料修訂版
- 服務運行秒數
- 伺服器時間

### 營運指標

```text
GET /api/metrics
```

會回傳：

- API 請求數
- API 錯誤數
- 成功資料異動數
- 客戶、保單、行程、團隊與任務筆數
- 記憶體使用量

這個端點目前已限制為主人或管理者；正式部署時可再交由內部監控系統定期擷取。

## 三、操作稽核

```text
GET /api/audit?limit=100
```

目前記錄：

- 建立、修改、刪除或整批同步
- 客戶、保單、行程或整體資料狀態
- 資料識別碼
- 修改前後版本
- 行程狀態
- 操作時間

稽核紀錄不寫入姓名、電話、Email、保單內容或服務備註。

目前已記錄操作者帳號與組織；正式營運仍需加入 IP 摘要與請求追蹤編號。

## 四、資料庫位置

預設資料庫：

```text
.data/sasha-workbench.sqlite
```

可透過環境變數變更：

```text
SASHA_DB_PATH=指定路徑
```

`.data`、備份檔及資料庫 WAL 檔已排除於版本控制之外。

### PostgreSQL 正式環境

1. 在部署主機執行 `npm install`，安裝專案鎖定的 `pg` 驅動。
2. 建立空白 PostgreSQL 資料庫及僅供工作台使用的帳號。
3. 設定 `SASHA_DATABASE_URL`，正式環境使用 `SASHA_DATABASE_SSL=verify-full`。
4. 執行 `npm run postgres:migrate`。遷移器會使用資料庫鎖，並核對每份 SQL 的 SHA-256。
5. 啟動後檢查 `/api/health` 的 `databaseEngine` 是否為 `postgresql`。

未設定 `SASHA_DATABASE_URL` 時，系統仍使用 SQLite。不可在同一正式環境交替使用兩種資料庫。

### SQLite 轉移至 PostgreSQL

1. 停止工作台，先建立並驗證一份 SQLite 備份。
2. 先預覽來源筆數：

```text
npm run postgres:import-sqlite -- --source=backups/備份檔.sqlite
```

3. 確認 PostgreSQL 是空白資料庫，再正式轉移：

```text
npm run postgres:import-sqlite -- --source=backups/備份檔.sqlite --confirm
```

轉移工具會在同一個 PostgreSQL 交易中匯入組織、帳號、MFA 復原碼、客戶、保單、行程、團隊、版本號及稽核紀錄，逐表比對筆數；任何錯誤都會整批回復。舊登入工作階段不會轉移，使用者須重新登入。工具會拒絕非空白目標，避免覆蓋正式資料。

若任何帳號已啟用 MFA，正式環境第一次啟動必須使用原本的 32 位元組主金鑰，否則無法解密既有 MFA 秘密。應先把原金鑰安全匯入秘密管理服務，確認登入成功後，再依未來的金鑰輪替程序重新加密。

### MFA 主金鑰

本機預設金鑰位置：

```text
.data/master.key
```

- 金鑰用於加密 MFA 秘密與雜湊復原碼，不會經由 API 傳回。
- 資料庫備份與金鑰應分開保存於不同的受控位置。
- 若只還原資料庫而沒有原金鑰，已啟用 MFA 的帳號將無法完成驗證。
- 正式環境應由秘密管理或 KMS 注入 `SASHA_MASTER_KEY`，內容須為 32 位元組的 base64 或 hex。
- 不可直接替換既有金鑰；目前尚未提供自動重新加密與金鑰輪替程序。
- 金鑰不得放入原始碼、聊天訊息、一般備份目錄或版本控制。

## 五、建立備份

先確認資料庫已存在，再執行：

```text
node scripts/backup.mjs
```

預設備份位置：

```text
backups/
```

備份流程會：

1. 建立 SQLite 一致性快照。
2. 重新開啟備份檔。
3. 執行 `integrity_check`。
4. 回報備份路徑、修訂版及完整性結果。

可以用 `SASHA_BACKUP_DIR` 指定其他備份資料夾。正式環境應將備份複製到加密且不同主機或不同雲端區域的位置。

## 六、驗證備份

```text
node scripts/verify-backup.mjs backups/備份檔.sqlite
```

驗證內容：

- SQLite 完整性
- 資料修訂版
- 各類資料筆數

備份檔必須定期抽樣驗證，不能只確認檔案存在。

## 七、還原備份

1. 先停止莎莎工作台伺服器。
2. 先執行不帶確認參數的預覽：

```text
node scripts/restore-backup.mjs --source backups/備份檔.sqlite
```

3. 確認來源與目標路徑後執行：

```text
node scripts/restore-backup.mjs --source backups/備份檔.sqlite --confirm
```

還原工具會：

- 先驗證來源備份。
- 為目前資料庫建立 `before-restore` 安全副本。
- 清除舊 WAL／SHM 暫存檔。
- 複製備份到正式資料庫位置。
- 重新執行完整性檢查。
- 若還原後驗證失敗，嘗試恢復原資料庫。

## 八、同步與衝突

- 客戶、保單與行程使用逐筆 API。
- 每筆資料有版本號。
- 舊版本修改會收到 `409 VERSION_CONFLICT`。
- 行程離線修改會進入本機待同步佇列。
- 其他本機修改會標記為尚待同步。
- 若離線期間伺服器已有新版本，系統不會自動覆蓋，會保留本機資料等待人工處理。

## 九、目前尚未達到的正式營運條件

- 經身分驗證的忘記密碼／帳號復原程序。
- 雲端 KMS、金鑰輪替及客戶敏感欄位加密。
- 受管理 PostgreSQL 實例的實際遷移、效能測試及災難回復演練。
- 集中式監控、告警及外部日誌服務。
- 自動排程備份及跨區備份。
- 法遵、個資及資安驗收。

在上述項目完成前，本版本仍應視為本機正式化開發版，不可直接公開到網際網路。
