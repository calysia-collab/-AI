## 階段 2 開發規格草案

本文件固定階段 2 規格。2026-06-13 本機正式環境驗證通過後已開始建置；實作進度請參閱 `階段2建置進度-2026-06-13.md`。

## 啟動條件

- GitHub Actions `verify` 與 `staging-gate` 全部通過。
- 受管理 PostgreSQL 完成遷移、備份及還原演練。
- KMS／秘密管理完成一次資料金鑰輪替。

- 有效 HTTPS、Secure Cookie、可信代理設定通過。
- Windows、macOS、iPhone、Android 驗收清單完成簽署。

## 階段 2 目標

1. 將客戶、保單、行程、團隊及附件改為完整領域資料模型。
2. 以版本化 REST API 取代整包 `/api/state` 寫入。
3. 建立資料匯入、匯出、查詢、分頁及封存機制。
4. 保留組織隔離、顧問指派、欄位加密與稽核紀錄。

## 預計資料模型

- `customer_profiles`：基本資料、聯絡偏好、來源、標籤、風險屬性。
- `customer_contacts`：電話、Email、地址及主要聯絡方式。
- `customer_relationships`：配偶、子女、家庭及企業關係。
- `policies`：保單主檔、保險公司、商品、狀態、繳費資訊。
- `policy_coverages`：保障項目、保額、保費、期間及附約。
- `policy_parties`：要保人、被保險人、受益人。
- `customer_interactions`：拜訪、電話、LINE、Email 與服務紀錄。
- `tasks`：負責人、期限、優先度、狀態與提醒。
- `documents`：附件、OCR 狀態、版本、保留期限及刪除狀態。
- `consents`：個資同意、行銷同意、版本、取得及撤回時間。

## API 原則

- 新 API 使用 `/api/v1`。
- 清單 API 必須支援游標分頁、排序及允許的篩選條件。
- 修改使用版本欄位或 ETag，避免覆蓋他人更新。
- 每筆寫入記錄 actor、organization、request ID 與最小必要稽核資訊。
- owner、manager、advisor、viewer 權限由共用授權層處理。
- `/api/state` 先改為唯讀相容端點，再於完成資料遷移後移除。

## 遷移策略

1. 新增 `004` 之後的不可變 PostgreSQL 遷移。
2. 建立舊資料轉換與筆數核對工具。
3. 先雙讀驗證，再切換新 API。
4. 確認資料、權限、稽核及效能後停止舊寫入。
5. 完成備份與回復演練後移除相容層。

## 驗收標準

- 所有新表都有 organization scope、必要索引及外鍵。
- 顧問只能看到自己被指派的客戶與相關資料。
- 敏感欄位在 PostgreSQL 中保持加密。
- 10 萬客戶測試資料下，主要清單查詢有分頁且不做全表載入。
- 匯入可預覽、驗證、取消；匯出有權限與稽核紀錄。
- API 契約、自動測試、遷移回滾方案與操作文件同步完成。
